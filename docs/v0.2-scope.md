# Lumo v0.2 Scope

Status: v0.2 seal-ready local control-layer scope.

## Goal

Lumo v0.2 is a local control layer that Codex can actively use during real
work.

The user gives Codex a normal request. Codex uses Lumo to choose the right
operating mode, gather the right context, continue autonomously inside safe
rails, and involve the user only when a real decision, risk, or proof boundary
appears.

```txt
User request
-> Codex
-> Lumo route / map / gate / review
-> Codex works
-> Lumo checkpoint / proof / learn
-> User sees only decision-worthy moments
```

## Tool Freeze

No new Lumo tools are added to v0.2 after this scope.

Finish only this set:

| Tool | Status | Role |
| --- | --- | --- |
| `doctor` | existing | Check local readiness before Lumo participates. |
| `analyze` | existing | Produce a repo rubric/report. |
| `init` | existing | Preview or write first repo-level harness files. |
| `preflight` | existing | Decide whether a task is sharp enough to start. |
| `checkpoint` | existing | Check whether active work is still on course. |
| `review` | existing | Decide whether Codex can honestly claim done. |
| `thread-checkpoint` | existing | Turn long thread evidence into a steering card. |
| `pr-status` | existing | Separate PR checks, findings, merge state, and next proof layer. |
| `harness-map` | local v0.2 | Map the user's actual AI-coding cockpit. |
| `route` | local v0.2 | Choose the operating mode and which Lumo tools Codex should use. |
| `learn` | local v0.2 light | Propose small harness improvements from repeated friction. |

Everything else is parked until after v0.2 dogfood:

- MCP server;
- TUI;
- SaaS;
- auth/license gate;
- multi-language expansion;
- Claude Code parity;
- automatic global rule rewriting;
- deep memory/session mining;
- full macOS app.

## Desired User Experience

Lumo should feel almost invisible to the user.

The user should not need to decide:

- whether to start a goal;
- whether to make a loop;
- whether to checkpoint;
- whether a PR is actually ready;
- whether Codex can claim done;
- which harness files or skills matter.

Codex should ask Lumo those questions. The user only sees Lumo when the answer
changes the next decision.

## Operating Modes

`route` is the front door for Codex.

It should classify a request into one of a few operating modes and recommend the
next Lumo tool.

| Mode | Use When | First Lumo Move |
| --- | --- | --- |
| `tiny_answer` | Explanation, rewrite, or safe local fact | No tool or direct answer |
| `lightweight_patch` | Small local code/docs change | `preflight` optional, then `review` |
| `standard_feature` | Multiple files, user-visible behavior, unclear route | `harness-map` if needed, then `preflight` |
| `bugfix_investigation` | Cause is unclear or reproduction matters | `preflight`, then checkpoint after evidence |
| `long_agent_thread` | Delegated/autonomous thread with growing context | `thread-checkpoint` |
| `pr_release` | PR, merge, release, deployment, runtime proof | `pr-status` |
| `harness_improvement` | Improving rules, skills, docs, or workflow | `harness-map`, then `learn` |

V1 can be deterministic and conservative. It does not need to be a perfect AI
router.

## Harness Map

Status: built as a local/read-only v0.2 command with markdown and JSON output,
tests, slice docs, and one dogfood case.

`harness-map` answers:

```txt
What does this agent already rely on?
```

It should map:

- global `AGENTS.md`;
- repo `AGENTS.md`, `CLAUDE.md`, `.cursor/rules`, and equivalent files;
- repo docs and workflow docs;
- package scripts and verification commands;
- available skills/plugins at a high level;
- obvious overlaps, gaps, and stale-looking rails.

It does not rewrite anything in v1. It inspects repo rails through the local
scan and only checks explicit or environment-provided Codex/Agents homes at the
metadata level.

## Route

Status: built as a local/read-only v0.2 command with markdown and JSON output,
tests, slice docs, and one dogfood case.

`route` is deterministic and conservative. It can return a silent surface for
tiny answer-only work, and it recommends `pr-status`, `thread-checkpoint`,
`harness-map`, `preflight`, `checkpoint`, `review`, or `learn` only when a local
control-layer step is useful.

## Learn Light

Status: built as a local/read-only v0.2 light command with markdown and JSON
output, tests, slice docs, and one dogfood case.

`learn` answers:

```txt
What small harness improvement follows from this friction?
```

It should accept a redacted packet or local run artifact and propose one small
change:

- add or adjust a repo rule;
- add a workflow note;
- add a deterministic check;
- update a skill prompt;
- do nothing because the friction was one-off.

It does not auto-write global rules, memories, skills, repo docs, GitHub,
Linear, CRM, production, or external systems in v0.2.

## Seal Criteria

Lumo v0.2 can be sealed when:

- `harness-map`, `route`, and `learn` exist as read-only/local commands;
- each has markdown and JSON output;
- each has tests;
- each has one dogfood case;
- README explains the Codex-facing flow in under two minutes;
- `npm test`, `npm run typecheck`, `npm run build`, and `npm run public:check`
  pass;
- the docs say clearly that v0.2 is a local control layer, not MCP/SaaS.

## What v0.2 Proves

v0.2 should prove:

```txt
Codex can use Lumo as a local second set of eyes during real work.
```

It does not prove:

- Lumo is production-grade;
- Lumo works for every stack;
- Lumo guarantees better code;
- Lumo replaces tests, review, or user judgment;
- MCP distribution is ready.

## v0.2 Seal Note

The v0.2 control-layer path is now local and dogfoodable:

```txt
route -> harness-map/preflight -> checkpoint -> review -> learn
```

The seal means the local commands exist, have markdown and JSON examples, have
unit/CLI coverage, and have one dogfood case each. It does not mean Lumo has an
MCP server, TUI, SaaS surface, package distribution story, clean-room Codex
isolation, external side effects, or production-readiness claim.
